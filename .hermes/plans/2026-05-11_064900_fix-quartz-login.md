# Fix Quartz 4 Blog Login Issue

## Goal
Diagnose and fix the login credential mismatch on the Quartz 4 blog so the user can access their private notes.

## Current Context

- **Platform**: Quartz 4 static site generator
- **Repo**: `/home/mtkagent/workspace/quartz-website/`
- **Authentication method**: Client-side password gate (NOT server-side)
- **Username configured**: `shibajahn`
- **Password verification**: SHA-256 hash comparison done in the browser (client-side only)

## Key Findings

### Auth Architecture
The authentication is **not** handled by Quartz itself. A custom component `quartz/components/PasswordGate.tsx` provides it:

```
quartz/
  components/
    PasswordGate.tsx          <-- Custom auth component (lines 7-9)
    pages/
      LandingPage.tsx         <-- Uses PasswordGate for index page
      Content.tsx             <-- Uses PasswordGate for index.md pages
    index.ts                  <-- Exports PasswordGate
```

### Current Credentials
- **Username**: `shibajahn` (normalized to lowercase on login)
- **Password hash**: `91b9db45eefb82bcab3ec0f77bc32c4f6ebeb5a7233c8e580c180522d636c8b7`

### Hash Mismatch
I tested the hash against many common passwords (`password`, `admin`, `shibajahn`, `qwerty`, `letmein`, `notes`, `private`, `password123`, `shibajahn123`, `shiba`, etc.) and **none matched**. The stored hash does not correspond to any commonly used password.

### How It Works
1. User visits the blog's homepage (index page)
2. `PasswordGate.tsx` renders a login form
3. User enters username + password
4. Browser computes SHA-256 of the password and compares to stored hash
5. If match, `sessionStorage.setItem("notes_auth", "true")` is set
6. The gate is hidden; the rest of the blog becomes visible
7. Internal navigation is intercepted to prevent access while unauthenticated

## Root Cause

The SHA-256 hash `91b9db45eefb82bcab3ec0f77bc32c4f6ebeb5a7233c8e580c180522d636c8b7` stored in `PasswordGate.tsx` does not match any password you may have tried. The hash was likely generated from a password that is no longer recalled, or was generated with a different tool/method than the expected `echo -n "password" | sha256sum | cut -d' ' -f1`.

## Proposed Approach

Choose one of the following options:

### Option A: Set a New Password (Recommended)
1. **Determine** the desired username and password
2. **Generate** the SHA-256 hash using: `echo -n "your_password" | sha256sum | cut -d' ' -f1`
3. **Update** the `USERS` dictionary in `quartz/components/PasswordGate.tsx` (line 8)
4. **Commit and deploy** the change (`git commit -am "update auth credentials" && git push`)

### Option B: Add Multiple Credentials
If you want multiple users or a fallback:
1. **Add** entries to the `USERS` dictionary with known passwords
2. **Keep** the old entry as a fallback until the new one is verified
3. **Commit and deploy**

### Option C: Temporary Debug Password
If you're unsure which password was used:
1. **Temporarily** add a known-good password to the `USERS` dictionary
2. **Verify** you can log in successfully
3. **Then** set the correct permanent password

## Security Considerations

- **⚠️ Client-side auth is not secure**: The password hash is stored in the JavaScript bundle visible to anyone viewing the page source. Anyone can:
  - Read the stored hash
  - Perform offline brute-force attacks against the hash
  - See the username `shibajahn` in plain text
  
- **This is acceptable** if the goal is casual protection (keeping casual visitors out), but **NOT acceptable** if the content is highly sensitive.

- **If stronger security is needed**, consider:
  - GitHub Pages deploy keys with password protection
  - Cloudflare Workers with basic auth in front of the static site
  - A serverless function that checks credentials before serving the site
